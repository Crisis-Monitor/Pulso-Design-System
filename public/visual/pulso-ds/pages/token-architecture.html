Foundations · Tokens

Token Architecture

Três camadas — primitive · semantic · component — separadas por intenção. Sem isso, rebrand é cirurgia. Com elas, é troca da camada 1 em um dia.

O problema que estamos resolvendo

Hoje misturamos primitivos (--lavender, --indigo-deep) com semânticos (--primary) e específicos de componente (--sidebar-accent) na mesma camada. Quando precisamos trocar a cor primária — rebrand, white-label, dark/light — cada componente vira refator manual.

Hoje · misturado
Component referencia primitive direto
Trocar #6C46F5 exige varredura por todo o repositório. Cada uso de "violeta" pode ser intencional (brand) ou acidental (decoração).
--button-bg: #6C46F5;
/* ou — pior — */
background: #6C46F5;
v2 · 3 camadas
Component → semantic → primitive
Trocar a cor primária = trocar uma referência na camada 1. As camadas 2 e 3 não sabem que houve mudança.
--button-primary-bg: var(--color-action-primary);
--color-action-primary: var(--violet-500);
--violet-500: #6C46F5;

As três camadas

Cada camada tem um papel claro. Componente nunca pula camada — sempre referencia a camada imediatamente acima.

01
Primitive
Valor literal. Sem significado. Sem opinião sobre uso.
~120 tokens
/* paleta crua */
--violet-500: #6C46F5;
--violet-700: #5532D6;
--gray-50:  #F5F4FA;
--gray-900: #0F0C1A;
--scale-4:  1rem;
--radius-2: 0.625rem;
Quem usa: apenas a camada semantic. Nunca componente, nunca produto.
02
Semantic
Intenção. Diz para quê serve, não o que é.
~80 tokens
/* propósito no produto */
--color-action-primary:
  var(--violet-500);
--color-surface-page:
  var(--gray-50);
--space-airy:
  var(--scale-4);
--radius-control:
  var(--radius-2);
Quem usa: camada component, e excepcionalmente produto quando não há componente.
03
Component
Aplicação. Específico de um componente ou família.
~160 tokens
/* aplicado a UI específica */
--button-primary-bg:
  var(--color-action-primary);
--card-bg:
  var(--color-surface-page);
--card-pad:
  var(--space-airy);
--input-radius:
  var(--radius-control);
Quem usa: CSS interno do componente. Produto nunca toca aqui.

Convenção de nomenclatura

Cada camada tem um padrão de nome. O nome diz a camada — não precisa ler a definição para saber se pode usar.

CamadaPadrãoExemplosPode usar em
L1 primitive --{family}-{step} --violet-500 · --gray-900 · --scale-4 · --radius-2 · --shadow-1 L2 apenas
L2 semantic --{kind}-{role}-{state?} --color-action-primary · --color-surface-page · --color-text-strong · --space-airy L3 + produto
L3 component --{component}-{slot}-{variant?} --button-primary-bg · --card-border · --input-focus-ring · --sidebar-item-active-bg CSS do próprio componente

Anti-patterns que a arquitetura previne

Padrões que vamos travar via lint rules + code review.

✕ Pular camada
Component referenciando primitive
Sintoma de que não existe semantic para essa intenção. Crie um.
--button-bg: var(--violet-500); /* errado */
--button-bg: var(--color-action-primary);
✕ Valor literal no produto
Hex inline em componente
Cor solta no JSX/SCSS. Lint bloqueia. Toda cor vem de token.
background: #6C46F5; /* errado */
background: var(--color-action-primary);
✕ Semantic com nome de família
"--color-violet-primary"
Carrega o hue no nome. Quebra na hora do rebrand. Use a função no nome, não a cor.
--color-violet-primary
--color-action-primary
✕ Primitive sintético
color-mix() em L1
L1 é valor literal estático. Misturas e cálculos vivem em L2 ou L3, onde a intenção justifica.
--violet-soft: color-mix(...);
--color-action-primary-soft: color-mix(in oklab, var(--color-action-primary) 14%, transparent);

Guardrails · lint rules

Documentação não basta. A v2 ativa três lint rules no eslint-plugin-pulso:

RuleBloqueiaNível
pulso/no-literal-color Hex, rgb(), hsl() em CSS-in-JS ou inline style. error
pulso/no-primitive-in-component Componente referenciando --violet-*, --gray-* diretamente. error
pulso/no-deprecated-token Token v1 marcado como deprecated. Mostra o substituto v2 no warning. warn → error em v3
