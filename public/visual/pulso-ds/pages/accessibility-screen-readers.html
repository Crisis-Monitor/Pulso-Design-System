Screen readers & ARIA contract

Operador com leitor de tela ainda gere crises. Esta página define o contrato ARIA por componente, a hierarquia de anúncios e o padrão de teste em VoiceOver + NVDA. Sem isso, um alerta crítico vira silêncio.

Hierarquia de anúncios · aria-live

Três níveis. Assertive interrompe a leitura corrente — reservado para crise. Polite espera silêncio. Off não anuncia (default). Errar de tier é tão ruim quanto não ter live region.

Tier 0
aria-live="off"
Default. Nada acontece. A maioria dos elementos vive aqui.
Exemplos: texto estático, cards de métrica em repouso, conteúdo de tab inativo.
Tier 1
aria-live="polite"
Anuncia depois da leitura atual. Não atropela o operador no meio de uma frase.
Exemplos: nova menção entra no feed, "filtro aplicado · 142 resultados", autosave concluído, snackbar de sucesso.
Tier 2
aria-live="assertive"
Interrompe imediatamente. Use só quando perder o anúncio = perder a crise.
Exemplos: alert critical, sessão expirou, ação destrutiva confirmada, threshold rompido.

Regra prática: se você está em dúvida entre polite e assertive, é polite. Assertive abusado vira ruído e o operador desliga o leitor.

Landmarks · esqueleto da página

Toda template do Pulso compõe a mesma estrutura. Leitor de tela navega entre landmarks com um atalho — operador chega ao conteúdo em duas teclas.

<!-- Esqueleto canônico de Crisis Detail -->
<a href="#main" class="skip-link">Pular para o conteúdo</a>

<header role="banner"> <!-- topbar do produto -->
  <nav aria-label="Workspace"> … </nav>
</header>

<nav aria-label="Navegação principal"> <!-- sidebar -->
  <a href="/dashboard" aria-current="page">Dashboard</a>
  …
</nav>

<main id="main" aria-labelledby="case-title">
  <h1 id="case-title">PLS-012 · Vazamento de dados</h1>
  <section aria-labelledby="sec-overview"> … </section>
</main>

<aside aria-label="Pulso IA"> <!-- painel lateral --> … </aside>
Únicos por página
banner, main, contentinfo (footer). Múltiplos nav e region são ok — desde que cada um tenha aria-label distinto.
Heading order
Um h1 por página · não pule níveis (h2 → h4 é proibido) · estrutura visual e estrutura de heading precisam coincidir.
Region vs section
Use <section> nativa com aria-labelledby. role="region" só quando o elemento é genérico (<div>) e a seção merece ser anunciada.

Skip link · saltar para conteúdo

Primeiro elemento focável de toda página. Invisível até receber foco — Tab com página recém-carregada o revela. Leva o foco para o <main>.

Texto exclusivo para leitor · .sr-only

Conteúdo invisível visualmente mas lido pelo SR. Útil quando o visual depende de cor/forma e o sentido precisa ser narrado.

/* Classe canônica · presente em globals.css */
.sr-only {
  position: absolute;
  width: 1px; height: 1px;
  padding: 0; margin: -1px;
  overflow: hidden;
  clip: rect(0, 0, 0, 0);
  white-space: nowrap;
  border: 0;
}

/* Uso típico — score com hint visual + label semântico */
<div role="status">
  <span aria-hidden="true">62</span>
  <span class="sr-only">Score 62 de 100, nível attention</span>
</div>

Estados de carga · aria-busy

Container que está carregando recebe aria-busy="true". Skeletons internos ganham aria-hidden="true" — leitor anuncia "ocupado" uma vez, não doze.

Container loading
<section aria-busy="true" aria-live="polite"> — quando terminar, atualizar o conteúdo e o leitor anuncia.
Skeleton individual
<div class="skeleton" aria-hidden="true"> — peças individuais são decoração.
Spinner standalone
<span role="status" aria-label="Carregando casos"> — sempre com label específico, nunca só "Carregando".
Página inteira
Aplique aria-busy="true" em <main>; o foco fica preservado e o leitor avisa quando concluir.

Regras

Faça
  • Use HTML semântico primeiro: <button>, <nav>, <main>, <dialog>. ARIA é remediar, não decorar.
  • Em alerta crítico, role="alert" (ou aria-live="assertive"). Em feedback rotineiro, polite.
  • Em qualquer IconButton, Switch ou Checkbox sem texto visível, aria-label é obrigatório.
  • Skip link como primeiro tab stop de toda página. <main id="main"> como destino.
  • Helper text e erro via aria-describedby / aria-errormessage — leitor lê na ordem certa.
Evite
  • Não use aria-live="assertive" para feedback rotineiro. Atropelar o operador queima a confiança no leitor.
  • Não use div onclick. Sempre <button>. Se precisar parecer link, estilize o botão — não inverta papéis.
  • Não esconda controles com aria-hidden="true" se eles ainda são focáveis. Inconsistência confunde mais que invisibilidade.
  • Não use placeholder como label. Some quando o usuário digita; SR pode não lê-lo.
  • Não escreva textos de SR longos demais ("Clique aqui para abrir o painel de configurações avançadas do caso"). Curto e direto: "Configurações do caso".
