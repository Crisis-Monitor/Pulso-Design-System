Matriz de estados

Loading, error e empty existem como componentes · faltava sistema. Toda tela deveria ter quatro versões — default, loading, empty, error — derivadas automaticamente. Esta página fixa o tier de loading, o tier de erro, o skeleton matching e o optimistic UI pattern.

Tier de loading · 4 níveis

Tempo de espera muda o tratamento. Instant não anuncia · Fast usa skeleton · Slow nomeia o escopo · Very slow mostra etapas.

TierJanelaTratamentoMicrocopy
01 · Instant < 100ms Direct render · sem skeleton · sem spinner Nenhuma
02 · Fast 100ms–1s Skeleton matching · sem texto · sem spinner Nenhuma
03 · Slow 1–3s Spinner inline + texto que nomeia o escopo "Buscando em 240 menções…"
04 · Very slow > 3s Progress bar + etapas nomeadas · revela "está vivo" "Lendo 42 menções · sintetizando narrativa · sugerindo resposta"

Tier de erro · 4 classes

Cada classe tem propósito distinto. Operador identifica em < 1 segundo se é responsabilidade dele (validation), da rede (network), do sistema (fatal) ou da política (permission).

networkrecuperável
Conexão perdida
Retentando em 3s · automático até reconectar.
Estado factual + plano automático + CTA manual. Nunca pede desculpa pelo que vai resolver sozinho.
validationuser fix
Threshold entre 0 e 100
Valor atual: −15 · default 75.
Constraint nomeado + valor atual + esperado. Nunca "por favor" — não é pedido de desculpa, é correção.
permissionpolicy
Apenas analistas declaram crise
Encaminhe para o analista de plantão ou peça promoção temporária ao gestor.
Qual papel pode + caminho para escalar. Nunca apenas "negado" — sempre nomeia autorizado e como pedir.
fatalsystem down
Caso indisponível agora
Rascunho persistido. Reabre automático quando voltar (≈ 4 min). ref: incident-2026-05-17-022
Estado + o que foi preservado + plano B + ref técnica. Reference number sempre — é o que o operador cola no Slack do suporte.

Empty states · 3 tipos

Vazio não é falha. Cada tipo tem tom próprio · primeira vez = orientação · após filtro = ajuste · fim do feed = encerramento.

first-timeonboarding
Nenhum caso aberto
Casos aparecem aqui quando o radar dispara ou um analista cria manualmente. Conecte fontes para começar.
Explica o que vai aparecer. Sempre tem 1 ação primária para tirar do vazio.
no-resultsfilter
Nenhum caso bate com esses filtros
Tente afrouxar o intervalo (atual: últimas 6h) ou remover o filtro de critical.
Sugere refinamento concreto, nomeando o filtro que provavelmente está apertado demais.
end-of-feedgood empty
Nada pede atenção agora
3 casos estão em stable. Próxima varredura do radar em 8 min.
Valida o estado · vazio aqui é bom · não é problema a resolver.

Skeleton matching · cada componente tem seu skeleton

Skeleton imita o layout final em block-level. Operador vê a tela "se formando" no lugar certo · sem layout shift quando os dados chegam.

ComponenteSkeleton blocksObservação
MetricCard1 label 70% · 1 valor 40% · 1 sub 60%Sem o delta chip — aparece quando o dado real chega
CrisisCard (lista)3 linhas: kicker · título 88% · meta 60%Mantém altura da row para evitar jump
MentionCardAvatar 32px · 2 linhas de texto · 1 footer 50%Skeleton avatar circular igual ao componente
ScoreRingCírculo cinza outline · sem fillMantém tamanho do donut
LineChartGrid + 1 linha cinza em zigue-zague suaveNão pisca · estático
Table rowMesmas colunas · cada cell um .skel-lineRender 6 linhas por padrão
Tab panelSkeleton do conteúdo da tab ativaOutras tabs não renderizam skeleton

Reduced-motion:  skeleton fica estático em var(--muted) · sem shimmer · ver 1.5.

Optimistic UI · ack, follow, dismiss

Para ações reversíveis e rápidas, o UI mostra o resultado antes da confirmação do servidor. Operador segue trabalhando · reversão acontece silenciosamente se o servidor falhar.

01 · click

Operador clica "ack" no alerta

UI atualiza imediatamente · contador de não-lidos cai · linha some da lista de alertas pendentes.

aplicado · otimista
02 · server confirma

Backend retorna 200 OK em 240ms

Estado já casava · nada visível para o operador. Audit log registra. Persistência confirmada silenciosamente.

confirmado
03a · ou erro

Backend retorna 5xx ou timeout

UI reverte com toast inline "Não foi possível confirmar ack · tentando reenviar…". 3 retries · depois mantém em "pendente" com aviso.

revertido

Regras

Faça
  • Toda tela tem 4 estados · default · loading · empty · error · derivados do mesmo componente, não 4 telas.
  • Use skeleton matching · não spinner full-page para listas com layout previsível.
  • Em erro fatal, inclua sempre request id · é o que o operador cola no chat do suporte.
  • Em loading slow, nomeie o escopo ("Buscando em 240 menções…"). Sem isso, parece travado.
  • Em ações destrutivas, nunca otimista. Operador precisa do "concluído" do servidor.
Evite
  • Não use spinner full-page por menos de 1s. Skeleton resolve · sem spinner sobreposto.
  • Não use "Carregando…" genérico em slow/very-slow. Nomear escopo é mandatório.
  • Não esconda completamente a UI durante loading. Manter cabeçalho + nav · só conteúdo carrega.
  • Não use otimista em ação de saída pública. Audit-critical sempre síncrono.
  • Não trate empty como erro. "Nada para mostrar" sem CTA é desorientador.
